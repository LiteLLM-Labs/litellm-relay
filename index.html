Default behavior: full proxy payload capture for configured AI domains, with redaction and Gateway ingestion.

v0 Acceptance Test

A user sends a prompt in Notion AI. Relay captures the decrypted request/response preview, redacts sensitive headers, sends an ingestion event to LiteLLM Gateway, and the event appears in the Gateway Logs UI.

How Gateway Ingests It

StepGateway behaviorWhy it works with Logs
1. Authenticate Relay calls POST /internal/collector/events with a Gateway virtual key or scoped ingest token. Gateway can attribute all local-machine captures to the right team, key alias, user, and org.
2. Normalize Gateway converts each Relay event into a SpendLogsPayload-compatible row. The existing Logs UI already knows how to list and expand LiteLLM_SpendLogs.
3. Persist Gateway inserts into LiteLLM_SpendLogs with duplicate skipping on request_id. No new table or new Logs UI surface is required for v0.
4. Display Logs reads it through GET /spend/logs/ui and detail reads through /spend/logs/ui/{request_id}. Relay rows appear beside normal LiteLLM Gateway requests, but labeled as captured local AI traffic.

Gateway Side

Gateway Logs already reads from LiteLLM_SpendLogs via /spend/logs/ui. For v0, Relay should call an authenticated ingestion endpoint that writes Relay rows into that same table. Synthetic /v1/chat/completions calls are only a smoke test; they create fake model calls and distort spend analytics.

Future Model Routing

After v0 proves capture and ingestion, Relay can optionally replace selected app calls with LiteLLM model calls. That is not needed for the first proof.

Keep v0 scoped to evidence: capture Notion AI payloads locally and show them in Gateway Logs. Routing Notion to LiteLLM models requires app-specific protocol adapters and response rewriting.
